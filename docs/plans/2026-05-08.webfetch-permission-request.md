# WebFetch Permission Request — Execution Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task-by-task.

**Goal:** Route ACP `WebFetch` permission requests through an inline Allow/Deny prompt rendered in the chat panel.

**Architecture:** Pure policy returns an `:awaiting-user` sentinel; `resolve-cb` in `main/effects/acp.cljs` becomes a Promise-returning seam that round-trips through IPC to the renderer; renderer holds ephemeral per-topic state, renders Allow/Deny inline (with orphan fallback), and replies via a synchronous `ipcMain.handle` RPC.

**Tech Stack:** ClojureScript, Electron IPC, Nexus state/actions, Replicant UI, Malli schemas, ACP SDK in-process bridge.

---

## Context

- ACP's `fetch`-kind permission requests today auto-allow `WebSearch` and reject everything else (`src/gremllm/schema/codec/acp/permission.cljs:100-109`). `WebFetch` is unusable.
- The branch `feat/web-fetch-request-user-permission` is the round-2 follow-up the existing comment anticipates.
- Pending permission requests live at `[:topics <id> :session :pending-permissions]` — co-located with `pending-diffs` for state-shape symmetry — but they are runtime-only and stripped by the topic persistence codec on save.

## Validation deltas vs. draft `docs/plans/2026-05-08.webfetch-permission-request.md`

The draft's high-level architecture stands. These are the corrections folded into the steps below:

1. **Ephemeral state must be excluded from the topic codec.** Pending-diffs is persisted; pending-permissions must not be. New schema entry stays under `[:topics <id> :session :pending-permissions]`; `src/gremllm/schema/codec.cljs` (where `topic-from-disk` and `topic-to-ipc` live — *not* `schema/codec/topic.cljs`) strips it on save. **Decision:** keep the path under `:session` rather than relocating to a `[:transient ...]` root; the codec filter is one `dissoc` and the symmetry with `pending-diffs` keeps the state model coherent.
2. **Message schema needs `:tool-call-id`.** `Message` in `src/gremllm/schema.cljs` doesn't carry it today; extend it. `start-response` and any tool-use message construction must thread it.
3. **WebFetch tool-call updates currently produce no renderer message.** Today only Read-with-file-metadata events emit a `:tool-use`. The orphan-rendering path is *load-bearing*, not just defensive — many WebFetch prompts will arrive before any tool-use message exists. We choose orphan rendering over extending `handle-tool-event` — simpler, fewer touch points, and it doubles as the IPC-ordering safety net.
4. **Main-side enrichment race — resolved.** `permission.cljs:25-34` documents the registry pattern as the protocol-level workaround: ACP delivers `tool_call` session updates before their `requestPermission`, and `tool-names` is seeded from `_meta.claudeCode.toolName` before resolution. No await loop needed; `enrich-permission-params` already handles the lookup idempotently. If `:tool-name` is genuinely missing the `case` falls through to reject — that's correct behavior.
5. **Response IPC: prefer `ipcMain.handle`.** The renderer→main response is a sync ack; main exposes `acp-effects/resolve-pending-decision!` which the IPC handler calls.
6. **JS bridge needs no change.** `src/js/acp/index.js:39-44` already `return`s the CLJS callback's value through `async`/`await`. Only the CLJS callback shifts to returning a `js/Promise`.
7. **Option validation lives in one place.** `effects/acp.cljs/await-user-decision` is the single source of truth for "do these options contain both an allow-* and a reject-*?" The `request-user-decision` injection in `core.cljs` is fire-and-forget IPC: it pushes the request and returns `(js/Promise.resolve nil)`. No duplicate validation.
8. **`:raw-input` is kebab-case after coercion.** Confirmed via `schema/codec/acp.cljs:264` (`[:raw-input {:optional true} AcpPermissionRawInput]`). The URL is read at `[:tool-call :raw-input :url]` — keyword path, not snake_case string path.

- [ ] **Step 3: Rewrite `resolve-cb`** (lines 142-155) to be Promise-returning:

```clojure
(defn- resolve-cb [^js raw-params session-cwd]
  (try
    (let [enriched (->> raw-params
                        acp-codec/acp-permission-request-from-js
                        (acp-permission/enrich-permission-params @tool-names))
          outcome  (acp-permission/resolve-permission enriched session-cwd)]
      (when on-permission
        (try (on-permission enriched) (catch :default e (js/console.warn "on-permission tap failed" e))))
      (if (= :awaiting-user (:outcome outcome))
        (-> (request-user-decision enriched)
            (.then (fn [option-or-cancel]
                     (acp-codec/acp-permission-outcome-to-js
                       (if (= :cancelled option-or-cancel)
                         {:outcome {:outcome "cancelled"}}
                         {:outcome {:outcome "selected" :option-id option-or-cancel}})))))
        (acp-codec/acp-permission-outcome-to-js outcome)))
    (catch :default e
      (js/console.error "resolve-cb failed" e)
      (acp-codec/acp-permission-outcome-to-js {:outcome {:outcome "cancelled"}}))))
```

Note: the synchronous arm still returns a JS object — the bridge accepts both because `await <non-promise>` is a no-op.

- [ ] **Step 4: Add registry helpers exposed for the IPC layer:**

```clojure
(defn resolve-pending-decision!
  "Looks up tool-call-id in :pending-decisions and resolves with option-id (or :cancelled).
   Returns true if a decision was waiting, false otherwise."
  [tool-call-id option-id-or-cancelled]
  (when-let [resolve-fn (get-in @state [:pending-decisions tool-call-id])]
    (swap! state update :pending-decisions dissoc tool-call-id)
    (resolve-fn option-id-or-cancelled)
    true))

(defn cancel-all-pending!
  "Resolves every waiting decision with :cancelled. Returns the set of tool-call-ids cancelled."
  []
  (let [pending (:pending-decisions @state)]
    (swap! state assoc :pending-decisions {})
    (doseq [[_ resolve-fn] pending] (resolve-fn :cancelled))
    (set (keys pending))))
```

- [ ] **Step 5: Build the `request-user-decision` injection-friendly helper** that registers the resolver inside the atom *before* invoking the shell-supplied `request-user-decision`:

```clojure
(defn- await-user-decision [request-user-decision enriched]
  (js/Promise.
    (fn [resolve _reject]
      (let [tool-call-id (get-in enriched [:tool-call :tool-call-id])]
        (swap! state assoc-in [:pending-decisions tool-call-id] resolve)
        (-> (request-user-decision enriched)
            (.catch (fn [e]
                      (js/console.error "request-user-decision threw" e)
                      (resolve-pending-decision! tool-call-id :cancelled))))))))
```

Then update the call site in `resolve-cb` to call `(await-user-decision request-user-decision enriched)` instead of `(request-user-decision enriched)`. This ensures the registry is populated before any IPC round-trip resolves.

### Task 5: Main IPC — push request, RPC response, push cancel

**Files:**
- Modify: `src/gremllm/main/core.cljs`

- [ ] **Step 1: Define `request-user-decision`** in `core.cljs`. By contract (established in Task 3), `await-user-decision` already registered the resolver in `:pending-decisions` and validated `has-allow-and-deny?` before this function is called. This function is **fire-and-forget IPC**: emit the push, return a no-op Promise. The resolver fires later via `resolve-pending-decision!` from the IPC response handler.

```clojure
(defn- url-from [enriched]
  ;; :raw-input keys are kebab-case after acp-permission-request-from-js
  ;; (confirmed in schema/codec/acp.cljs:264).
  (get-in enriched [:tool-call :raw-input :url]))

(defn- option-id-of-kind [options kinds]
  (some (fn [opt] (when (kinds (:option-id opt)) (:option-id opt))) options))

(defn request-user-decision
  "Pushes acp:permission-request to renderer. Resolver is already registered
   by effects/acp.cljs/await-user-decision; this is the IPC emit."
  [enriched]
  (let [options      (:options enriched)
        allow-option (option-id-of-kind options #{"allow_once" "allow_always"})
        deny-option  (option-id-of-kind options #{"reject_once" "reject_always"})]
    (ipc-effects/send-to-renderer "acp:permission-request"
      #js {:acpSessionId  (:acp-session-id enriched)
           :toolCallId    (get-in enriched [:tool-call :tool-call-id])
           :toolName      (get-in enriched [:tool-call :tool-name])
           :url           (url-from enriched)
           :allowOptionId allow-option
           :denyOptionId  deny-option})
    (js/Promise.resolve nil)))
```

`has-allow-and-deny?` validation is **not** duplicated here — Task 3 already short-circuits to `:cancelled` when options are malformed, so this function only runs against valid input. Keeping option validation in one place (`await-user-decision`) is the design discipline.

- [ ] **Step 3: Implement `permission.cljs`:**

```clojure
(ns gremllm.renderer.actions.permission
  (:require [gremllm.renderer.state.topic :as topic-state]))

(defn- pending-path [topic-id]
  [:topics topic-id :session :pending-permissions])

(defn received
  "Handler for :permission.events/received. Stores entry by tool-call-id under the
   topic owning acp-session-id."
  [state {:keys [acp-session-id tool-call-id] :as entry}]
  (if-let [topic-id (topic-state/topic-id-by-acp-session state acp-session-id)]
    [[:state.effects/assoc-in (conj (pending-path topic-id) tool-call-id) entry]]
    ;; Unknown session — drop silently (could log).
    []))

(defn respond
  "Handler for :permission.actions/respond. Sends RPC and dissocs on success."
  [state tool-call-id option-id]
  (let [topic-id (:active-topic-id state)]
    [[:effects/promise
      {:promise    (.acpPermissionRespond js/window.electronAPI
                     #js {:toolCallId tool-call-id :optionId option-id})
       :on-success [[:state.effects/dissoc-in (conj (pending-path topic-id) tool-call-id)]]
       :on-error   [[:state.effects/dissoc-in (conj (pending-path topic-id) tool-call-id)]]}]]))

(defn cancelled
  "Handler for :permission.events/cancelled. Dissocs the entry without RPC."
  [state tool-call-id]
  ;; Cancellation arrives on whichever topic the prompt belongs to — search topics.
  (when-let [topic-id (topic-state/topic-id-with-pending-permission state tool-call-id)]
    [[:state.effects/dissoc-in (conj (pending-path topic-id) tool-call-id)]]))
```

(`topic-state/topic-id-by-acp-session` and `topic-state/topic-id-with-pending-permission` are small lookups added in Task 4. Add them now if missed.)

## Resolved during validation

These were open during draft review and are settled here so execution doesn't restate them:

1. **ACP ordering — confirmed.** `permission.cljs:25-34` documents the registry pattern as the protocol-level workaround: the agent emits `tool_call` session updates before the corresponding `requestPermission`, and `_meta.claudeCode.toolName` seeds the `tool-names` registry. `enrich-permission-params` resolves the name idempotently. **No await loop needed in main.** The renderer-side orphan-rendering pass (Task 9) handles the *IPC delivery* race between the `acp:permission-request` push and the `tool_call` update flowing through to the renderer's chat state — a different ordering concern at a different layer.
2. **Topic codec location — confirmed at `src/gremllm/schema/codec.cljs`.** `topic-from-disk` is at line 10; `topic-to-ipc` at line 37; `topic-from-ipc` at line 41. There is no `src/gremllm/schema/codec/topic.cljs` and there should not be one — the existing module already owns these concerns.
3. **`raw-input :url` shape — kebab-case keyword path.** Confirmed at `schema/codec/acp.cljs:264` (`[:raw-input {:optional true} AcpPermissionRawInput]`). After `acp-permission-request-from-js` coerces wire-format JS into CLJS, all keys are kebab-case keywords. The path is `[:tool-call :raw-input :url]`. No string-key fallback needed.
4. **Persistence path for `pending-permissions` — keep under `[:topics <id> :session :pending-permissions]`.** Considered moving to a `[:transient ...]` root to physically prevent persistence, but the codec filter is one `dissoc` and the `:session`-co-located shape mirrors `pending-diffs`, which keeps the renderer state model coherent (one accessor pattern, one mental model). Trade-off accepted: relies on codec discipline rather than structural impossibility.
